geminiとの議論用に、DriveAccesserプロジェクトのコードマップを以下に示す。

0,前提
geminiは、実際のコードを確認しながら議論を行うこと。
架空のコードを基に話されては、活用できる情報が少なすぎる。
記憶できる文字数も発言できる文字数もわずかなのだから、無意味な情報を膨らませるのは絶対に避けたい。



1,目的と概要
DriveAccesserプロジェクトは、ストレージとのアクセスを担当するライブラリである。
要点は二つ。
・ストレージへのアクセスを抽象化し、様々なストレージサービスに対応できるようにすること。
現状はローカルファイルシステムとGoogleDriveへの対応を目指している。
・アクセス制限を管理し、許可のないアクセスを防止すること。
許可のないアクセスは全てエラーを返すようにする。



2,基本的な概念と対応するクラス

・パス文字列→DriveItemPathとその派生クラス
ローカルファイルシステムであれば、絶対フルパスを表す文字列をvalueとする。
GoogleDriveであれば、ドライブアイテムのIDを表す文字列をvalueとする。

*その他も引数・プロパティとして文字列型は使わず、enumやrecordの利用を徹底している。

・アクセス権限→個別権限のFileSystemPermissionと、複合権限のFileSystemPermissionBundle
FileSystemPermissionBundle._rootに記述されているFileSystemPermission群の外にあるアクセスは全てエラーを返す。

・ストレージへのアクセスメソッド
個別権限のFileSystemPermissionに対応する、IDriveAccesserとその派生クラス。
複合権限のFileSystemPermissionBundleに対応する、UseDriveAccesserとMultiDriveAccesser。



3,想定される利用法
ソリューション外のストレージへアクセスする必要がある場合に、DriveAccesserをクローンし、
そのソリューションにおいて必要なアクセス権限を記述する。

その上で、固定パスへのアクセスを行う場合はUseDriveAccesserを継承し、ストレージの存在すら意識せず処理できるメソッドを製作する。
動的なパスへのアクセスを行う場合はMultiDriveAccesserを継承し、パスを受け取って処理するメソッドを製作する。



4,データの流れと構造

・パス文字列/DriveItemPath
機能:
文字列からの生成
親ディレクトリのDriveItemPathの取得
そのパスの存在確認
利用箇所：
DriveItemInfoのプロパティ
FileSystemPermissionのプロパティ
実際のアクセスを行うメソッドの引数

・ファイル情報/DriveItemInfo
プロパティ：
DriveTypeEnum DriveType
DriveItemPath Path
string Name
FileSystemType FileType //拡張子やMIMEタイプなどに対応するenum
bool IsDirectory
long? Size = null
DateTime? LastModified = null
GoogleDriveMetadata? Metadata = null // GoogleDrive の時だけセットされる
機能：
System.IO.FileInfoからの生成
System.IO.DirectoryInfoからの生成
GoogleDriveMetadataからの生成
利用箇所：
情報確認用メソッドの返り値

・グーグルドライブのメタデータ/GoogleDriveMetadata及びGoogleDriveMetaDataBank
グーグルAPIの消費を減らすために、GoogleDriveのアイテムの情報をキャッシュするためのクラス。
プロパティ：
GoogleDrivePath Id,
string Name,
FileSystemType Type,
long? Size,
GoogleDrivePath? ParentId = null,
string? ETag = null
機能：
拡張メソッドにより、GoogleItemPath.InBank()でキャッシュ済みかの確認と、
GoogleItemPath.FromBank()でGoogleDriveMetadataへの変換が可能。
利用箇所：
DriveItemInfoのプロパティ
GoogleItemPath.Parent及びGoogleItemPath.existsに使用

・個別権限型/FileSystemPermission
プロパティ：
DriveTypeEnum DriveType //LocalDrive、GoogleDriveなどのドライブ種を表すenum
DriveItemPath Path
PermissionScope Scope　//自身、子ファイル、孫以降ファイルの3種のFlagEnum
FileSystemAccessLevel AccessLevel //Read、Append、Create、Delete、Writeの5種のFlagEnum
FileSystemType FileType //拡張子やMIMEタイプなどに対応するenum
機能：
指定のDriveItemPathをアクセス範囲に含むかどうかの確認
指定のFileSystemPermissionに自身が含まれるかどうかの確認
利用箇所：
FileSystemPermissionBundle._rootに記述される個別権限の型
他は、内部処理としてのみ利用されることで、FileSystemPermissionBundle._rootに存在しない権限でのアクセスを回避している。

・複合権限型/FileSystemPermissionBundle
フィールド：
原版として使う静的フィールドである_root
実際の権限を保持するインスタンスフィールドである_permissions
機能：
空権限かどうか
単独権限かどうか
個別権限を返すAsSinglePermission()
他の個別権限もしくは複合権限を含むかどうかの確認
一部の個別権限を取り出した複合権限を返すGetPart()
個々の個別権限を小さく変更し、空権限を除いた複合権限を返すNarrow()
AccessLevel以外全て同じ個別権限を統合するMergeAccessLevel()
利用箇所：
IDriveAccesser及びUseDriveAccesserの生成

・個々のアクセスメソッド/IDriveAccesserとその派生クラス
機能：
実際のストレージへのアクセス各種
利用箇所：
GoogleItemPath.Existsの内部処理
GoogleItemPath.Parentの内部処理

・固定パスへのアクセス用の基底クラス/UseDriveAccesser
機能：
内部処理用に、指定した権限のみでアクセスするIDriveAccesserを生成するメソッドを提供
利用箇所：
外部で使用する前提。

・不特定パスへのアクセス用の基底クラス/MultiDriveAccesser
機能：
IDriveAccesserと同様のメソッドを、DriveItemPathを追加の引数とする形で提供する。
利用箇所：
外部で使用する前提。




ex1,改善予定

FileSystemPermissionBundle._rootにハードコードしている原版を、jsonなどに外部化した方がいいのではないか。

FileSystemPermission.IsItemAllowed()は唯一のDriveItemInfoを引数型とするメソッドだが、これは適正なのか。

GoogleItemPath.parentにもforce引数を追加して、キャッシュを無視してGoogleDriveAPIを呼び出すオプションを追加した方がいいのではないか。

上位権限と指定権限から具体的なaccesserを生成するメソッドがFileSystemPermissionBundleに存在しているが、個々のDriveAccesserに持たせるべきではないか。
というか、FileSystemPermissionBundleはDrive種が増えたことで書き直されてはならないと思う。

適当に配置したエラー型は適切か否か。


ex2,指摘された改善点
AccessGoogleDriveファイル上のあれこれは、DriveAccesserを前提とした構造を作り始める前のコードであるため、DriveAccesserとの整合性が無いのは当然とする。
GoogleDriveAccesserは現状何の実装もなされていない。
GoogleDriveAccesserを作り始める前に、既存部分の修正と、GoogleDriveを扱うために必要なクラスが十分かの確認を行う必要がある。

IDriveAccesser系列に関しては、全部internalで大丈夫っぽい……？
→基底のIDriveAccesserのみpublicで公開、実装クラスはinternalにするため、トリプルディスパッチの前段をdynamicで踏みつぶす形で修正中。
